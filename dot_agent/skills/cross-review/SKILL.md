# Cross Review（Composer 2.5 + Codex）

既存の計画書・実装 diff・PR などを対象に、**自己レビュー**と **Codex レビュー**を並列実行し、指摘を突合・統合して人間が最終判断するワークフロー。報告は日本語で行う。

## 前提

- レビュー対象は **既に存在する成果物**（計画書、実装 diff、PR 本文など）。本スキルは作成ではなくレビューに専念する
- **最終採否は人間が決める**。エージェントは推奨と根拠を提示するだけ
- Codex の指摘は鵜呑みにしない。実コードと公式ドキュメントで裏取りする

### 前提となるツール

| ツール | 確認 | 用途 |
|--------|------|------|
| `codex` CLI | `codex --version` | Codex 並列レビュー（[skill-codex](../skill-codex/SKILL.md) 準拠） |
| `gh` CLI | `gh --version` | PR 情報取得・diff・レビュー投稿 |

## 厳守する制約

- **レビュー中は修正しない**。コードの修正・パッチ適用は、人間が「修正して」と明示した場合のみ（Step 6）
- **PR へのインラインコメント／レビュー投稿は、必ず `AskQuestion` で明示承認を得てから行う**。無断投稿は禁止
- 投稿する場合も `event` は **`COMMENT` 固定**（`APPROVE` / `REQUEST_CHANGES` はしない＝マージ可否の最終判断は人間に委ねる）
- **PR 投稿用の本文では Composer / Codex のどちらの指摘かは明示しない**（統合した 1 つのレビューとして提示）。チャット内の報告書（Step 5）では仕分けを残す
- PR レビュー用の **一時 worktree・一時ファイルは最後に必ず後始末**する
- Codex が利用制限・エラーで失敗した場合は、その旨を伝え **自己レビュー結果のみ**で統合・報告する

## ワークフロー概要

```
[確定] レビュー対象の特定 + Codex モデル選択
    ↓
[PR のみ] レビュー用 worktree 準備
    ↓
[並列] ① 自己レビュー（Composer 2.5）  ||  ② Codex レビュー
    ↓
[統合] 指摘の仕分け + Codex 固有の裏取り
    ↓
[提示] レビュー報告書 → 人間が採否決定
    ↓
[PR のみ・任意] 承認後に PR へ COMMENT 投稿
    ↓
[修正] 採用指摘の反映（ユーザー指示時）
    ↓
Major 以上の指摘が残存 かつ 人間の「このまま進む」指示なし
    → Step 0 から再実行（サイクル N+1） ──┐
    ↑                                    │
    └────────────────────────────────────┘
    ↓
Major 以上ゼロ、または人間が未採用のまま実装続行を指示 → 完了
    ↓
[PR のみ] worktree・一時ファイルの後始末
```

## Step 0: レビュー対象の確定

レビュー開始前に以下を明確化する:

| 項目 | 例 |
|------|-----|
| 対象種別 | 計画書 / 実装 diff / PR / 複合 |
| スコープ | ファイル一覧、ブランチ、PR 番号 |
| 基準ブランチ | `main` など（diff レビュー時） |
| 重点観点 | セキュリティ、設計、テスト不足 など（ユーザー指定があれば） |
| Codex モデル | 実行時に `codex debug models` で調査して確定（Step 2 参照） |

対象が PR の場合:

1. `gh pr view <PR> --json number,title,state,isDraft,baseRefName,headRefName,headRefOid,author,additions,deletions,changedFiles` を取得
2. `state` が closed、または `isDraft` が true の場合はその旨を伝えて中断（必要性をユーザーに確認）
3. `baseRefName` が `main` / `develop` 以外なら **スタック PR**。base 差分の取り違えに注意する
4. リポジトリは `gh repo view --json nameWithOwner -q .nameWithOwner` で解決（URL 指定時は owner/repo/番号 をパース）
5. PR 本文・diff は `gh pr view` / `gh pr diff` で取得

実装レビューは `git diff [base]...HEAD` を基準とする。計画書の場合は対象テキストまたはファイルパスを特定する。

**Codex レビューを開始する前に**、Step 2 のモデル選択ポリシーに従いモデルと reasoning effort を確定する。並列レビュー（Step 1 / Step 2）は、モデル確定後に開始する。

### PR レビュー用 worktree 準備（Codex 用）

PR を対象とする場合、Codex 実行前に **必ず** detached worktree を用意する。`codex exec` はローカルの git state を対象とするため、PR 番号を直接渡せない。

```bash
git fetch origin <headRefName> <baseRefName>
git worktree prune
git worktree add --detach /tmp/pr<番号>-review origin/<headRefName>
git -C /tmp/pr<番号>-review rev-parse HEAD   # headRefOid と一致することを確認
```

- **このリポジトリは worktree が複数あり、PR ブランチが既に別 worktree でチェックアウト済みのことが多い。** 現 worktree で checkout すると競合するため、必ず **detached worktree を新規作成**する
- Codex の `-C` は `/tmp/pr<番号>-review` を指定する
- レビュープロンプトでは `origin/<baseRefName>...HEAD` の差分に限定する旨を明記する
- 自己レビュー（Step 1）は `gh pr diff` または worktree 内の diff を参照してよい

## Step 1: 自己レビュー（Composer 2.5）

**レビュー対象を第三者視点で客観的に評価する。** 作成者の意図や文脈に引きずられないよう、以下を守る。

### 客観化のルール

1. **作成者モードを切る**: 「なぜこう書いたか」ではなく「第三者が読んで何が問題か」で見る
2. **チェックリストを機械的に適用**: 感情や意図を入れず、事実ベースで指摘する
3. **重大度を付ける**: Critical / Major / Minor / Nit
4. **根拠を必ず書く**: ファイルパス・行番号・該当コードを引用する

### 自己レビュー チェックリスト

- [ ] 要件・計画との整合性（抜け・過剰実装）
- [ ] 正確性（ロジックバグ、境界条件、エラーハンドリング）
- [ ] セキュリティ（入力検証、認可、秘密情報）
- [ ] 設計（責務分離、既存パターンとの一貫性）
- [ ] テスト（不足ケース、意味のあるアサーション）
- [ ] 可読性・保守性（命名、複雑度、コメントの妥当性）
- [ ] 計画書の場合: 実装可能性、リスク記載、スコープ明確さ

### 出力形式（自己レビュー）

各指摘をこの形式で記録する（後の仕分け用 ID を付与）:

```markdown
### [C-001] {タイトル}
- **重大度**: Critical | Major | Minor | Nit
- **場所**: `path/to/file.ts:42`
- **指摘**: {何が問題か}
- **根拠**: {コード引用または仕様との乖離}
- **修正案**: {具体的な改善案}
```

ID プレフィックス `C-` = Composer 自己レビュー。

## Step 2: Codex 並列レビュー

Step 1 と **並列**で Codex に同じ対象をレビューさせる。Codex の起動は [skill-codex](../skill-codex/SKILL.md) に従う。**PR の場合は worktree 準備完了後、バックグラウンド（Shell `block_until_ms: 0` または Task）で起動し、自己レビューと待ち時間を重ねる。**

### Codex モデル選択ポリシー

Codex レビュー実行前に、**ユーザーが選択したモデル**で `codex exec` を実行する。モデルは依頼時に必ず確定させる。**モデル一覧はスキル内に固定しない。** 使用時点の Codex CLI が返す最新カタログから生成する。

#### 1. 利用可能モデルを調査する

`AskQuestion` で選択肢を出す前に、必ず以下を実行してモデルカタログを取得する:

```bash
codex debug models 2>/dev/null
```

出力は JSON。`models` 配列から **ユーザーに提示する候補**を次の条件で絞る:

- `visibility` が `"list"`（UI に表示されるモデル）
- `supported_in_api` が `true`

各候補から以下を抽出し、選択肢一覧を組み立てる:

| フィールド | 用途 |
|-----------|------|
| `slug` | `-m` に渡すモデル ID |
| `display_name` | 選択肢ラベル |
| `description` | 選択肢の補足説明 |
| `default_reasoning_level` | reasoning 未指定時の既定 |
| `supported_reasoning_levels[].effort` | そのモデルで選べる reasoning effort |

**推奨マーク（Recommended）** は、`~/.codex/config.toml` の `model` と一致する slug に付ける。config に未設定、または一致モデルが候補にない場合は、slug に `codex` を含むモデルを優先し、なければ `priority` が最も高い候補を推奨とする。

調査失敗時（コマンドエラー・JSON 解析失敗・候補 0 件）:

1. ユーザーに失敗を報告する
2. `~/.codex/config.toml` の `model` / `model_reasoning_effort` をフォールバック候補として提示する
3. ユーザー確認後に実行する（無確認での実行はしない）

#### 2. 依頼時に動的一覧を提示する

ユーザーがモデルを明示していない場合、**調査で生成した一覧**を `AskQuestion` の選択肢として使い、**モデル**と **reasoning effort** を **1 回の質問セット（2 問）** で選ばせる。

- **モデル**: 調査結果の `slug` 一覧（`display_name` + `description` をラベルに含める）
- **reasoning effort**: 選択されたモデルの `supported_reasoning_levels` から生成（モデルごとに選べる値が異なる場合がある）

固定のモデル名・reasoning 値をスキル内にハードコードしない。

#### 3. ユーザー指示を優先する

- 依頼文でモデルが指定されている場合（例: 「gpt-5.4 でクロスレビュー」）→ そのモデルを使用。`AskQuestion` は省略可
- 指定モデルが調査結果に存在するか確認する。存在しない場合は最新カタログを再取得し、ユーザーに確認してから実行
- reasoning effort も指定されていればそれを使用。未指定ならそのモデルの `default_reasoning_level` を使用（カタログに無い場合は config の `model_reasoning_effort`、それも無ければ `high`）

#### 4. 確定後に `codex exec` を実行する

選択（または指定）されたモデル・reasoning effort を `-m` と `--config` に反映して実行する。**レビュー用途の既定サンドボックスは `read-only`**。

```bash
codex exec --skip-git-repo-check \
  -m "{選択されたモデル}" \
  --config model_reasoning_effort="{選択された effort}" \
  --sandbox read-only \
  -C "{リポジトリの絶対パス}" \
  "{レビュー用プロンプト}" </dev/null 2>/dev/null
```

- PR の場合 `-C` は `/tmp/pr<番号>-review`
- `--skip-git-repo-check` は必須（skill-codex 準拠）
- `</dev/null` は必須（stdin ブロック防止）
- `2>/dev/null` は既定（thinking tokens 抑制）
- 同期実行を優先（skill-codex の timeout 表を参照）。PR 並列時はバックグラウンド起動可

報告書の「対象」セクションに、使用した Codex モデルと reasoning effort を記録する。

### レビュー用プロンプトテンプレート

```text
以下をコードレビューしてください。指摘のみ返してください（修正は不要）。

## レビュー対象
{計画書 / ブランチ diff / PR の説明}

## リポジトリ
{絶対パス}

## 差分スコープ（PR の場合）
origin/{baseRefName}...HEAD の差分のみを対象とする

## 重点観点
{あれば記載}

## 出力形式
各指摘を以下の形式で:
- 重大度: Critical | Major | Minor | Nit
- 場所: file:line
- 指摘: ...
- 根拠: ...
- 修正案: ...
```

### Codex 実行の既定値

| 項目 | 既定 |
|------|------|
| サンドボックス | `read-only` |
| モデル | `codex debug models` で調査 → ユーザー選択（未指定時は `AskQuestion`） |
| reasoning effort | 選択モデルの `supported_reasoning_levels` から生成 → ユーザー選択（未指定時は `default_reasoning_level`） |
| 実行 | 同期実行を優先（skill-codex の timeout 注意） |
| stdin | `</dev/null` を必ず付与 |

Codex の出力にも ID を付与: `X-001`, `X-002` …（`X-` = Codex）。

## Step 3: 指摘の仕分け

両方のレビューが揃ったら、Composer 2.5 が以下の 3 カテゴリに分類する。

### 分類定義

| カテゴリ | 定義 |
|----------|------|
| **共通指摘** | 両者が同じ問題を指摘。または本質的に同一の懸念 |
| **Codex 固有** | Codex のみが指摘。Composer は確認したが問題なし、または見落とし |
| **Composer 固有** | Composer のみが指摘。Codex は言及なし |

### マッチング基準

「同一指摘」とみなす条件（いずれか）:

- 同じファイル・行（±数行）の同じ論点
- 同じバグ・設計問題を異なる表現で指摘
- 同じ欠落（テスト不足、エラー処理など）を指摘

曖昧な場合は **別指摘として両方残し**、仕分けメモに「関連あり」と書く。無理に統合しない。

重複指摘は 1 つにまとめ、片方だけの指摘も取りこぼさない。confidence の低いもの・false positive は落とす。必要に応じて worktree 内または対象ファイルを Read し、指摘の妥当性を再確認する。

### 仕分け表テンプレート

```markdown
## 指摘仕分け

| ID | 出典 | カテゴリ | 重大度 | 場所 | 要約 | 裏取り状態 |
|----|------|----------|--------|------|------|------------|
| C-001 / X-003 | 両方 | 共通 | Major | api/foo.ts:10 | 入力未検証 | 確認済 |
| X-002 | Codex | Codex固有 | Major | bar.ts:5 | 非推奨 API 使用 | 要検証 |
| C-002 | Composer | Composer固有 | Minor | baz.ts:20 | 命名が不明瞭 | — |
```

## Step 4: Codex 固有指摘の裏取り

**Codex 固有の指摘は鵜呑みにしない。** Composer 2.5 が以下の手順で検証する。

### 裏取り手順

1. **実コードを読む**: 指摘箇所の前後を Read ツールで確認。Codex が挙げた行・シンボルが実在するか検証
2. **公式ドキュメントを確認**: WebSearch / WebFetch で該当 API・ライブラリの公式情報を参照
3. **プロジェクト文脈を確認**: `.cursor/rules/`、既存パターン、テストから意図的な判断でないか確認
4. **判定を付ける**:

| 判定 | 意味 |
|------|------|
| **妥当** | コード・ドキュメントで裏付けられ、修正を推奨 |
| **部分妥当** | 論点は正しいが重大度・範囲が過大/過小 |
| **誤指摘** | コード・ドキュメントで反証。採用不要 |
| **要判断** | 根拠が割れる。人間に判断を委ねる |

### 裏取り記録テンプレート

```markdown
### X-002 裏取り結果
- **Codex の主張**: {要約}
- **コード確認**: {実際のコード状況}
- **ドキュメント**: {公式 URL と要点}
- **判定**: 妥当 | 部分妥当 | 誤指摘 | 要判断
- **Composer の見解**: {推奨対応と理由}
```

誤指摘と判断した場合も、なぜ誤りかを簡潔に記録する（後から検証可能に）。

## Step 5: レビュー報告書の提示

人間が採否を決められるよう、以下の構成で報告する。

```markdown
# クロスレビュー報告

## 対象
- 種別: {計画書 / 実装 / PR #N}
- スコープ: {ブランチ・ファイル}
- Codex モデル: {モデル名}（reasoning: {effort}）
- レビューサイクル: {N} 回目
- レビュー日: {日付}

## サマリー
- Critical / Major 指摘: N 件（Critical X / Major Y）
- 共通指摘: N 件
- Codex 固有: N 件（妥当 X / 誤指摘 Y / 要判断 Z）
- Composer 固有: N 件
- 前回からの変化: {新規 / 解消 / 未解消の ID 一覧}

## 総評（マージ可否の所見）
{統合した所見。出典は明示しない}

## actionable な指摘（`path:line` 付き）
{採用検討すべき指摘の一覧}

## 共通指摘（最優先で確認推奨）
{各指摘の詳細 + 修正案}

## Codex 固有指摘（裏取り済み）
{指摘 + 裏取り結果 + Composer 推奨}

## Composer 固有指摘
{各指摘の詳細}

## 推奨アクション
| 優先 | ID | 推奨 | 理由 |
|------|-----|------|------|
| 高 | C-001/X-003 | 修正すべき | 両者一致 + セキュリティ |

## 人間の採否（要記入）
以下についてご判断ください:
- [ ] {ID}: 採用 / 却下 / 保留 — メモ:

## サイクル判定
- [ ] 採用指摘を修正した → 再レビューへ
- [ ] Major 以上の指摘なし → 完了
- [ ] 未採用の指摘が残るが、このまま実装に進む
```

## Step 5b: PR への投稿（承認必須・PR のみ）

対象が PR で、ユーザーが PR 上へのレビュー投稿を望む場合のみ実施する。

1. `AskQuestion` で「この内容を PR にインラインコメント付きレビューとして投稿してよいか」を確認する（選択肢: 投稿する / 投稿しない）
2. **承認された場合のみ**:
   - 投稿本文は Step 5 の **総評** と **actionable な指摘** を統合し、**Composer / Codex の出典は含めない**
   - インラインコメントの `line` は **新ファイル側の行番号**（追加行は `side: "RIGHT"`）。`gh pr diff` で行番号を確認してから指定する（ずれると API が 422）
   - payload JSON を `/tmp` に作成:
     - `commit_id`: `headRefOid`（PR head の full sha）
     - `event`: `"COMMENT"`
     - `body`: 総評
     - `comments`: `[{ "path", "line", "side": "RIGHT", "body" }, ...]`
   - `gh api --method POST /repos/<owner>/<repo>/pulls/<番号>/reviews --input <payload.json>`
   - 投稿後 `gh api /repos/<owner>/<repo>/pulls/<番号>/comments` でインラインが付いたか検証する
3. **否認の場合は投稿せず**、報告書をチャットに提示して終了する

投稿名義は実行ユーザーの `gh` 認証アカウント。

## Step 6: 修正後の再レビューサイクル

採用された指摘をレビュー対象に反映したあとは、**必ず再レビューを実施する。** 1 回の報告で終了しない。

### サイクルの流れ

1. Step 5 で報告書を提示し、人間が各指摘の採否を決める
2. 人間が「修正して」等と指示した場合、**採用された指摘のみ**修正する
3. 修正完了後、**更新されたレビュー対象**に対して Step 0〜5 を再実行する（サイクル N+1）
4. 以下の **終了条件** のいずれかを満たすまで 2〜3 を繰り返す

### 終了条件（いずれかで完了）

| 条件 | 意味 |
|------|------|
| **Major 以上の指摘がゼロ** | Critical / Major の指摘が報告書に存在しない |
| **人間が続行を指示** | 提示した指摘を全て採用せず、このまま実装・マージに進む旨を明示 |

「Minor / Nit のみ残存」は終了条件を満たす。人間が続行を指示した場合、残存する Major 以上の指摘があってもサイクルを終了する（未解消指摘は報告書に記録して引き継ぐ）。

### 再レビュー時のルール

- **対象は修正後の最新状態**を使う（diff・PR・計画書を再取得）
- **前回の採否結果を引き継ぐ**: 却下済み ID は再指摘されても「却下済み」と記録。採用済み ID は修正確認の対象
- **Codex モデル**: 同一セッション内の再レビューでは前回と同じモデル・reasoning effort を使用（変更指示がなければ `AskQuestion` を省略）
- **報告書にサイクル番号**と前回からの変化（解消 / 新規 / 未解消）を必ず記載
- 再レビュー開始前に、前回採用指摘の修正が完了していることを確認する
- PR 再レビュー時は worktree を作り直し、`headRefOid` を再確認する

### エージェントの振る舞い

- 修正完了後、人間の追加指示を待たず **再レビューへ進む**（「再レビューします」と宣言して Step 0 から開始）
- Major 以上が残存し、かつ人間の続行指示がない場合、**自動的に次サイクルへ入る**
- 人間が「このまま進めて」「未採用で OK」等と指示した場合のみサイクルを終了する

---

## Step 7: 後始末（PR のみ）

PR レビューで worktree を作成した場合、**完了時（投稿の有無を問わず）** に以下を実行する:

```bash
git worktree remove /tmp/pr<番号>-review    # 必要なら --force
```

- 一時 payload JSON を削除
- `git worktree list` で対象 worktree が消えたことを確認する

---

## 採否のルール

- **エージェントは修正を自動適用しない**（ユーザーが「修正して」と明示するまで）
- 共通指摘でも、人間が却下すれば却下。エージェントは異議を述べず記録する
- 「保留」は理由を確認してから再レビュー
- 採用された指摘のみ、別タスクとして実装する
- **採用指摘の修正後は Step 6 に従い再レビューを実施する**（Major 以上が解消するか、人間が続行を指示するまで）

---

## トリガーと使い分け

| シナリオ | 動作 |
|----------|------|
| 計画書レビュー | Step 0〜5。Codex に計画書テキストを渡す |
| 実装 diff レビュー | diff を両者に渡す。裏取りは実コード中心 |
| PR レビュー | Step 0（worktree 準備）→ 1〜5 → 5b（投稿は承認時のみ）→ 7（後始末） |
| レビュー依頼時 | ユーザーが対象を指定した時点で Step 0 から開始 |
| 修正後の再レビュー | Step 6 に従い、Major 以上が解消するか続行指示まで Step 0〜5 を繰り返す |

---

## 注意事項

- 自己レビューと Codex は **並列**で開始し、待ち時間を最小化する
- Codex が失敗した場合: 自己レビュー結果のみで報告し、Codex 再実行を提案。仕分けは Codex 分を空欄にする
- 自己レビューで見落としがあった場合、Codex 固有 → 裏取り「妥当」なら素直に認める
- 共通指摘は最も信頼度が高いが、それでも人間の最終判断が優先
- 再レビューサイクル中は、却下済み指摘の再提示を最小限にし、新規指摘と未解消の Major 以上を優先して報告する
